Part 1: Understanding DNS

What is DNS? (Simple Explanation)

DNS = Domain Name System is like the internet's phone book. It translates human-friendly domain names into computer-friendly IP addresses.

üîë Key Concept

You type: learningmyway.space

DNS converts to: 34.118.224.132 (example IP)

Your browser: Connects to the IP address and loads the website

Why Do We Need DNS?

Without DNS: You'd need to remember 34.118.224.132 to visit your website

With DNS: You just type learningmyway.space (easy to remember!)

Detailed Record Explanations

A Record (Address Record)

Real Example for learningmyway.space:

Name: @
Type: A
TTL: 300
Data: 34.118.224.132

What it does: Points your domain to an IPv4 address

Use cases:

  • Main website (learningmyway.space)
  • Subdomains (vpn.learningmyway.space)
  • Multiple IPs for load balancing

CNAME Record (Canonical Name)

Real Example:

Name: www
Type: CNAME
TTL: 3600
Data: learningmyway.space.

What it does: Creates an alias pointing to another domain

Use cases:

  • www pointing to root domain
  • blog.learningmyway.space ‚Üí myblog.ghost.io
  • External services (CDN, hosting)
‚ö†Ô∏è CNAME Restrictions

Cannot use CNAME for:

  • Root domain (@) - Use A/AAAA instead
  • If other records exist for that name (MX, TXT, etc.)

MX Record (Mail Exchange)

Real Example (Google Workspace):

Name: @
Type: MX
TTL: 3600
Priority: 1
Data: aspmx.l.google.com.

What it does: Specifies mail servers to handle email

Priority: Lower number = higher priority (try first)

Multiple MX records: Provides redundancy if primary fails

TXT Record (Text)

SPF Example:

Name: @
Type: TXT
TTL: 3600
Data: "v=spf1 include:_spf.google.com ~all"

What it does: Stores arbitrary text (most versatile!)

Common uses:

  • SPF - Email sender authorization
  • DKIM - Email signature
  • DMARC - Email policy
  • Domain verification (Google, Facebook, etc.)
  • Site ownership proof

CAA Record (Certificate Authority Authorization)

Real Example:

Name: @
Type: CAA
TTL: 3600
Flags: 0
Tag: issue
Value: "letsencrypt.org"

What it does: Controls which CAs can issue SSL certificates

Why important: Prevents unauthorized certificate issuance

üí° Tags:
  • issue - Authorize certificate issuance
  • issuewild - Authorize wildcard certificates
  • iodef - Report violations to email

    Part 7: Troubleshooting

    Common Issues & Solutions

    ‚ùå Issue 1: Domain Not Resolving

    Symptoms:

    Browser shows "This site can't be reached" or "Server not found"

    Troubleshooting Steps:
    1. Check nameservers are correct:
      $ dig learningmyway.space NS +short
      Should show: ns-cloud-c1.googledomains.com
    2. Check A record exists:
      $ dig learningmyway.space A +short
      Should show: 34.118.224.132
    3. Check from Google's DNS:
      $ dig @8.8.8.8 learningmyway.space A
    4. Wait for propagation: Up to 48 hours, usually 2-4 hours
    5. Clear local DNS cache:
      $ # Windows
      ipconfig /flushdns

      $ # Mac
      sudo dscacheutil -flushcache

      $ # Linux
      sudo systemd-resolve --flush-caches

    ‚ùå Issue 2: WWW Not Working (but root domain works)

    Solution:

    Add CNAME record for www:

    DNS Name: www.learningmyway.space
Type: CNAME
TTL: 300
Data: learningmyway.space.

    ‚ùå Issue 3: Email Not Receiving

    Checklist:
  • MX records exist: dig learningmyway.space MX
  • SPF record configured
  • Email service (Google Workspace/Zoho) activated
  • Check with email provider's diagnostic tool

    • ‚ùå Issue 4: Changes Not Taking Effect

    Reason: TTL & Caching

    DNS changes are cached for the TTL period. If you had TTL=3600 (1 hour), changes take up to 1 hour to fully propagate.

    Solution:
    1. Before making changes: Lower TTL to 300 (5 min)
    2. Wait for old TTL to expire
    3. Make your changes
    4. After changes propagate: Raise TTL back to 3600

    Check propagation:

    $ dig @8.8.8.8 learningmyway.space A
